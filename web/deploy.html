Sección 08

Deploy y publicación:
tu trabajo, visible para todos

⏱️ Duración estimada: 20 min
📖 Concepto

¿Qué es el deploy?

Deploy significa publicar. Es el acto de tomar lo que tenés en tu computadora y dejarlo accesible en internet para que cualquier persona pueda verlo con un link.

"Tu compu es el taller. GitHub es la bodega. Render es la vidriera. Vos trabajás en el taller y decidís cuándo poner algo en la vidriera."

Hoy en IT4W usamos tres piezas que se conectan en cadena:

💻 Tu compu — donde editás y guardás cambios
↓ git push
GitHubGuarda el historial de versiones en la nube
↓ automático
RenderDetecta el cambio y publica la versión nueva
↓
🌐 URL pública — accesible para cualquier persona
💡 El flujo del día a día

Desarrollo vs. producción

Estas palabras suenan técnicas pero el concepto es simple:

🔧 Desarrollo (tu compu)
Donde hacés los cambios. Nadie más lo ve. Podés romper cosas, probar, deshacer. Es tu espacio de trabajo.
🌐 Producción (la URL pública)
Lo que ven todos. Solo llega acá lo que vos decidís publicar con un push. Es la versión "oficial".

El flujo completo de cada cambio que hacés:

PasoQué hacésQué pasa
1. EditásCambiás un archivo en VS CodeEl cambio existe solo en tu compu
2. Commitgit commit -m "descripción"Git guarda una foto del estado actual
3. Pushgit pushSube esa foto a GitHub
4. DeployRender lo detecta soloLa URL pública se actualiza en ~1 min
💡 Vos solo hacés los pasos 1, 2 y 3. El 4 pasa solo — Render vigila GitHub las 24 horas.
🛠️ Ejercicio — Publicá tu proyecto

Tu página personal, en vivo en internet

Durante toda la guía fuiste construyendo tu proyecto mi-proyecto-ia: creaste el index.html, le agregaste secciones, definiste tu estilo y generaste contenido. Ahora lo vas a publicar con una URL que podés compartir con cualquiera.

✅ ¿Qué tenés en mi-proyecto-ia a esta altura?
  • index.html — tu página con nombre, área y sección "Sobre mí"
  • comparacion.md — tus notas de la comparación Copilot vs Claude
  • sobre-mi.md — el texto que generaron los subagentes
  • .vscode/instructions/ — tu skill de tono personalizado
  • .github/copilot-instructions.md — tu agente principal
  • Git inicializado con historial de cambios

Panel de chat de VS Code — modo Agent
Guardá el estado final del proyecto
Un commit con todo lo que construiste a lo largo de la guía

Abrí mi-proyecto-ia en VS Code si no lo tenés abierto. En el chat en modo Agent, copiá esto:

💬 Copiá esto en el chat — modo Agent 
Hacé un commit con todos los archivos del proyecto. El mensaje tiene que decir "Versión final lista para publicar".

El agente agrega todos los archivos y guarda el punto final. Si Git no estaba inicializado, el agente también lo inicializa.

2
GitHub — navegador
Creá un repo en GitHub y subí tu proyecto
Tu proyecto en la nube, listo para conectar a Render

Abrí github.com, iniciá sesión y creá un repo nuevo:

1. Click en "New" (botón verde arriba a la izquierda)
2. Nombre del repo: mi-proyecto-ia
3. Dejalo en Public (o Private si preferís)
4. No marques ninguna opción de "Add README" — tiene que quedar vacío
5. Click en "Create repository"

GitHub te va a mostrar la URL del repo. Copiala — tiene el formato https://github.com/tu-usuario/mi-proyecto-ia.git

Ahora conectá tu carpeta local con ese repo. Pedile al agente:

💬 Copiá esto en el chat — modo Agent (reemplazá la URL) 
Conectá este proyecto con el repo de GitHub https://github.com/TU-USUARIO/mi-proyecto-ia.git y hacé un push.

El agente ejecuta los comandos para conectar y subir todo. Reemplazá la URL con la tuya real.

💡 Si GitHub te pide usuario y contraseña, ingresalos.
3
Render — navegador
Conectá Render con tu repo de GitHub
El servicio que publica tu página automáticamente

Abrí render.com — si no tenés cuenta, creá una gratuita (podés usar tu cuenta de GitHub para registrarte, es lo más rápido).

1. Click en "New +" → "Static Site"
2. Conectá tu cuenta de GitHub si te lo pide
3. Elegí el repo mi-proyecto-ia
4. Name: mi-proyecto-ia (o lo que quieras)
5. Branch: main
6. Publish directory: . (un punto solo — porque el index.html está en la raíz)
7. Click en "Create Static Site"

Render va a tardar 1-2 minutos en hacer el primer deploy. Vas a ver un log con texto corriendo — cuando aparezca "Deploy live", terminó.

4
Navegador
Abrí tu URL pública y verificá que funciona
Ver tu página publicada — y entender el flujo completo

Render te muestra la URL al final de la pantalla — algo como https://mi-proyecto-ia.onrender.com. Abríla y deberías ver tu página con tu nombre y todo lo que le fuiste agregando a lo largo de la guía.

✅ Si ves tu página: Completaste la guía entera. Cada sección que recorriste construyó algo que ahora está publicado en internet. Podés compartir ese link con quien quieras.
🕐 Si no cargó todavía: Esperá 2-3 minutos y recargá con Ctrl + Shift + R. Render puede tardar un poco en el primer deploy.

Bonus: Hacé un cambio en el index.html (cambiá una palabra), pedile al agente que haga commit y push, y en 1-2 minutos vas a ver el cambio reflejado en la URL. Ese es el flujo del día a día.

Fin del ejercicio ✓
✅ Checkpoint

¿Qué aprendiste en esta sección?

  • 🌐 Deploy es publicar tu trabajo en una URL accesible para todos.
  • 🔧 Desarrollo es tu compu — donde trabajás sin que nadie vea.
  • 📦 Commit es guardar una foto del estado de tu proyecto con Git.
  • ⬆️ Push es subir esa foto a GitHub.
  • 🤖 Render detecta cada push y actualiza la URL pública automáticamente.
  • 💬 El agente puede hacer el commit y el push por vos — solo pedíselo.
